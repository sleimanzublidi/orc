# Task Selection — 2026-04-11  (Iteration 2)

## Debate Summary

Verified all six ideas against the codebase. Key findings:

- **Database indexes (Eng #1):** Confirmed — zero secondary indexes in `MigrationManager.swift` (v1 migration, lines 10-62). The concurrent-run guard at `WorkflowEngine.swift:209-214` does loop through 3 statuses with 3 separate `listRuns(status:)` calls. Claims are technically accurate. However, this is a project-scoped SQLite database for a CLI tool — tables will have dozens to low hundreds of rows. The claimed 15-25ms → <1ms improvement is real in theory but imperceptible to users at this scale. This is a "correct" optimization, not a "needed" one.
- **Decompose `executeNestedWorkflow` (Eng #2):** Confirmed — method spans lines 578-844 (267 lines) with 7+ repetitive `do { try await store.updateNodeExecution(...) } catch { logger.warning(...) }` blocks. The refactoring is sound, but it's purely internal. No user can tell whether this method is 30 lines or 300 lines. The "testability" argument has merit, but decomposing for testability is premature when the existing integration tests cover the end-to-end paths. The iteration 1 fix already converted the `try?` calls in this method to proper `do/catch` logging, reducing the urgency.
- **ProviderRegistry protocol gap (Eng #3):** Confirmed — `ProviderRegistry` is a concrete `public struct` referenced by 12 files across Engine. The protocol gap is real per the project's stated convention. But `ProviderRegistry` has exactly one method (`provider(named:)`) and is already `Sendable`. The "testing ergonomics" benefit is marginal — Engine tests already use `FakeAgentProvider` injected into a real `ProviderRegistry`, and that works fine. The provider execution helper extraction is speculative — need to measure actual duplication before committing to it.
- **Interactive input prompting (Prod #1):** Real UX pain point. `TemplateError.unresolvedVariable` deep in execution is a terrible first-run experience. But the idea bundles TWO features (interactive prompting + `--input-file`). Interactive prompting requires TTY detection, readline/prompt rendering, and graceful degradation for non-interactive contexts. `--input-file` alone would be a clean single commit. Bundled together, this is too large.
- **Dry-run preview (Prod #2):** Strong idea. Reuses the existing `ExecutionPlanner`, `WorkflowParser`, and `TemplateResolver`. The key insight: at dry-run time, you can resolve input-based templates but NOT upstream-output-based templates — so the preview shows the plan structure, parallelization groups, and which variables are resolvable vs. pending. This is exactly what workflow authors need during the write-run-debug cycle. Implementation: add `dryRun()` to `WorkflowEngine` (parse → plan → resolve what's possible → return), plus formatting in CLI. 100-200 lines of new code, purely additive.
- **Rich terminal status (Prod #3):** Ambitious but way too large. DAG visualization with box-drawing characters, color coding, `--watch` mode with periodic refresh, critical path highlighting — this is weeks of work, not a single commit. The simpler version (just `--watch` on existing `orc status`) is achievable but that's not what's proposed.

Scoring criteria: product ideas advancing the product direction score highest on Value. Engineering ideas score high only if they deliver measurable user-facing benefit. At current scale (CLI tool, small databases, few users), internal refactors score 1-2 on Value.

## Selected Task

### Execution Plan Preview (`orc start --dry-run`)
**Source:** Product Ideas #2
**Priority:** P1
**Score:** Value 5 × Feasibility 4 × Safety 4 = **80**

**Description:** Add a `--dry-run` flag to `orc start` that resolves the full execution plan without dispatching any nodes. The output shows: the resolved DAG with execution order and parallelization groups (nodes at the same topological level run concurrently), template variables resolved where possible (inputs and builtins are resolved; upstream output references are shown as `<pending: {{node_id.output}}>`), `when:` guard expressions (shown as-is since most reference upstream outputs not yet available), and any validation warnings. This gives workflow authors instant feedback on their DAG structure without waiting for execution.

**Why this wins:**
- **Highest composite score (80):** Beats all engineering ideas on Value (5 vs 1-2) and ties or beats on Feasibility/Safety. The engineering ideas solve problems users can't see; this solves a problem workflow authors hit every time they iterate on a YAML file.
- **Directly advances the product:** Every serious workflow tool (Airflow, GitHub Actions, Terraform) has plan/preview functionality. This fills a foundational gap in the workflow authoring experience. The current feedback loop is: write YAML → run → wait minutes → discover issues at runtime. Dry-run collapses this to milliseconds.
- **Reuses existing infrastructure:** `WorkflowParser.parse()`, `ExecutionPlanner.plan()`, and `TemplateResolver.resolve()` already exist. The implementation adds a new `WorkflowEngine.dryRun()` method that orchestrates them without dispatching to providers. CLI formats the result. No new dependencies, no schema changes.
- **Safe and additive:** New code path, doesn't modify existing execution behavior. The `--dry-run` flag is a standard CLI pattern. If the formatting isn't perfect in v1, it's easy to iterate.

**Implementation notes:**
1. Add `dryRun(workflowFile:inputs:) async throws -> DryRunResult` to `WorkflowEngine`. This method: parses the workflow, validates inputs, builds the execution plan, and for each node attempts template resolution against the provided inputs + builtins (workspace path can be a placeholder like `<workspace>`). Unresolvable variables (upstream outputs) are left as tagged placeholders.
2. Define `DryRunResult` in Models: contains the plan's topological levels (each level is a group of node IDs that would run concurrently), per-node detail (resolved prompt/command or partial resolution with placeholders, `when:` expression, loop config, dependencies), and any validation warnings.
3. Add `--dry-run` flag to `StartCommand` in CLI. When set, call `engine.dryRun()` instead of `engine.start()`, format the result, print, and exit.
4. Output format: tree or table showing execution order, with nodes grouped by parallelization level. Each node shows its agent, resolved prompt snippet (truncated), dependencies, and `when:` guard if present. Validation warnings at the end.
5. No schema changes, no migration, no store interactions needed.
