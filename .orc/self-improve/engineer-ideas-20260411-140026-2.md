## 1. Add Missing Database Indexes to Store Schema

**Impact:** High
**Risk:** Low

**Description:** The SQLite schema (`Core/Store/Source/MigrationManager.swift`, lines 10-62) defines three tables â€” `runs`, `node_executions`, and `logs` â€” with zero secondary indexes. Every query that filters by a non-primary-key column performs a full table scan:

- `node_executions` is queried by `run_id` in `getNodeExecutions(runID:)`, by `(run_id, status)` in `getAwaitingInput(runID:)`, and by `(run_id, node_id)` in the resume path. No index on any of these columns.
- `logs` is queried by `node_execution_id` in `getLogEntries(nodeExecutionID:)`. No index.
- `runs` is queried by `status` in `listRuns(status:)` (called 3 times sequentially for the concurrent-run guard in `WorkflowEngine.start()`, lines 209-213). No index on `status`.

Add a v2 migration that creates indexes: `(run_id)` and `(run_id, status)` on `node_executions`, `(node_execution_id)` on `logs`, and `(status)` on `runs`. GRDB's migrator handles forward-only migrations cleanly â€” a new `"v2"` block in `MigrationManager.migrate()` is all that's needed.

A secondary optimization: the concurrent-run guard calls `store.listRuns(status:)` three separate times for `.running`, `.pending`, `.awaitingInput`. Adding a `listRuns(statuses:)` method to `WorkflowStoring` that accepts a `Set<RunStatus>` would collapse this to a single `WHERE status IN (...)` query, eliminating two round-trips.

**Rationale:** This is the highest-leverage, lowest-risk performance improvement available. Every `orc status`, `orc logs`, `orc list` command and the engine's own dispatch-loop queries hit these unindexed paths. For workflows with loop nodes (which can generate hundreds of `node_execution` records per run), the cost is linear in table size. The fix is purely additive â€” a new migration block, no behavioral changes, no API surface changes.

**Baseline estimate:**
- *Current:* `orc logs <run-id>` on a workflow with 50 node executions and 200 log entries: ~15-25ms per query (full scan of both tables). `orc start` concurrent-run guard: 3 sequential queries, ~5ms each = ~15ms overhead before every run.
- *Expected:* Index lookups: <1ms per query. Concurrent-run guard with batch query: single query <1ms.
- *How to measure:* Enable SQLite query tracing via GRDB's `Configuration.prepareDatabase { db in db.trace { print($0) } }` in a debug build. Run `orc logs` on a run with many executions. Compare wall-clock query times before and after indexes. Alternatively, use `EXPLAIN QUERY PLAN` on the raw SQL to confirm index usage.

## 2. Decompose `executeNestedWorkflow` and Extract Error-Recording Helper

**Impact:** High
**Risk:** Medium

**Description:** `NodeDispatcher.executeNestedWorkflow` (`Core/Engine/Source/NodeDispatcher.swift`, lines 578-844) is 267 lines â€” the longest method in the codebase by a factor of 3x. It handles 8 distinct concerns in sequence: YAML parsing, input resolution, workspace creation, execution planning, child run creation, child DAG dispatch, output extraction, and cleanup. Each concern has its own `do/catch` block that creates or updates a `NodeExecution` record with `.failed` status and logs a warning if the store write fails. This same error-recording pattern is duplicated 8+ times in this method alone, and appears in `executeSingleNode` and `executeLoopNode` as well (lines 342-358, 406-419, 536-561).

Proposed refactoring in two steps:

**(a) Extract an error-recording helper.** Replace the repeated pattern:
```swift
do { try await store.updateNodeExecution(..., status: .failed, errorMessage: ...) }
catch { logger.warning("Failed to persist ...") }
```
with a private method: `recordNodeFailure(executionID:nodeID:runID:error:store:logger:) async`. This eliminates ~80 lines of boilerplate across `NodeDispatcher`.

**(b) Decompose `executeNestedWorkflow` into focused private methods.** Each concern becomes a private method that returns a result or throws: `parseNestedWorkflow(...)`, `resolveNestedInputs(...)`, `createNestedRun(...)`, `dispatchNestedDAG(...)`, `extractNestedOutput(...)`. The parent method becomes a ~30-line orchestrator that calls each step and uses the shared error-recording helper for failure paths.

**Rationale:** This method is the primary testability barrier in the Engine module. Testing any specific failure path (e.g., "what happens when the nested workflow's YAML is invalid?") currently requires constructing a full `NodeDispatcher` with mocked store, parser, template resolver, providers, and evaluator â€” then triggering the exact condition at the right point in a 267-line method. After decomposition, each concern can be tested independently with minimal setup. The error-recording helper also prevents subtle bugs: the current pattern has minor inconsistencies (some paths create a new `NodeExecution` record, others update an existing one; some include the node ID in the log message, others don't).

**Baseline estimate:**
- *Current:* Cyclomatic complexity of `executeNestedWorkflow` is ~22 (8 do/catch blocks, each with an inner conditional). Adding a new nested-workflow feature (e.g., parameterized inputs from the roadmap) requires modifying a 267-line method.
- *Expected:* Orchestrator method complexity ~5. Each extracted method complexity ~3-4. New features add a new step method without touching the orchestrator's control flow.
- *How to measure:* Count lines and branches before/after. Verify test count: the decomposed version should enable 5+ focused unit tests for individual steps that are currently impossible to write in isolation.

## 3. Close the `ProviderRegistry` Protocol Gap and Extract Provider Execution Helper

**Impact:** Medium
**Risk:** Medium

**Description:** The project's CLAUDE.md states: "Protocol-first boundaries at every layer crossing; concrete types stay `internal`." `ProviderRegistry` (`Core/Providers/Source/ProviderRegistry.swift`) violates this â€” it is a `public struct` that crosses from Providers into Engine, where it is stored and passed through `NodeDispatcher`, `LoopHandler`, `InteractiveHandler`, and `EvaluatorRunner` as a concrete type. This is the only significant protocol boundary violation in the codebase.

Additionally, all three providers (`ClaudeCodeProvider`, `ShellProvider`, `CLIAgentProvider`) duplicate ~30 lines each for: temp file creation for stdout/stderr capture, non-zero exit code error construction, and interactive-mode delegation to `TmuxProviding`. This boilerplate means adding a new provider requires copying and adapting ~30 lines of infrastructure code.

Proposed changes:

**(a) Define a `ProviderRegistering` protocol** in Models with a single method: `func provider(for agentName: String) -> (any AgentProviding)?`. Make `ProviderRegistry` conform to it. Update Engine to depend on `any ProviderRegistering` instead of the concrete struct. This also enables testing Engine components with a minimal fake registry (just a closure-backed struct) rather than constructing a full `ProviderRegistry` with real or fake providers.

**(b) Extract a `ProviderExecutionHelper`** (internal to Providers) that encapsulates temp file creation, process execution via `ProcessRunning`, exit-code checking, and stdout/stderr reading. Each provider's `execute()` becomes a ~10-line method that builds the command arguments and delegates to the helper. This reduces per-provider code by ~60% and ensures consistent error handling.

**Rationale:** The protocol gap matters because it prevents testing Engine dispatch logic without importing the full Providers module. Currently, `FakeAgentProvider` in Engine tests is injected *into* a real `ProviderRegistry`, which means Engine tests construct a real Providers-module type. A `ProviderRegistering` protocol would let Engine tests use a trivial fake registry. The provider duplication matters for extensibility: the custom CLI agent support (roadmap-adjacent) means more provider types are likely, and each one currently requires copying the same boilerplate.

**Baseline estimate:**
- *Current:* Each provider implementation: ~80-100 lines. Adding a new provider: copy ~30 lines of boilerplate infrastructure. Engine test setup: must construct real `ProviderRegistry` with fake providers.
- *Expected:* Each provider: ~40-50 lines. New provider: ~20 lines (build args, delegate to helper). Engine test setup: construct a closure-backed fake registry in 3 lines.
- *How to measure:* Line count per provider before/after. Count of `ProviderRegistry` references in Engine module before/after (should drop to zero, replaced by protocol). Count of test setup lines in `WorkflowEngineTests` and `NodeDispatcherTests`.
