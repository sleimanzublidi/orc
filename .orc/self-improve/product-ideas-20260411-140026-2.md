# Product Ideas — 2026-04-11  (Iteration 2)

## 1. Interactive Input Prompting When Required Inputs Are Missing

**Impact:** High
**Effort:** Medium
**Description:** When a user runs `orc start workflow.yml` without providing all `required: true` inputs, instead of failing with an error, Orc should interactively prompt for each missing input. Display the input name, type, and description (from the workflow YAML `input:` block) as the prompt label. If a `default:` value exists, show it in brackets and accept Enter to use it. For non-interactive contexts (piped stdin, CI), fall back to the current error behavior. Additionally, support an `--input-file params.yml` flag that loads inputs from a YAML dictionary, enabling reusable parameter sets for workflows that take many inputs. This eliminates the pain of typing long chains of `--input key=value` flags and reduces the error surface for first-time users.

**Rationale:** The current experience when inputs are missing is a cryptic `TemplateError.unresolvedVariable` deep in execution (identified in Iteration 1's error messages idea but not implemented). Interactive prompting solves this at the root: users don't need to memorize input names or read the YAML file before running a workflow. This is the same pattern used by `gh pr create`, `npm init`, and `docker run -it` — CLI tools that gracefully degrade from flags to prompts. It also advances the roadmap's "parameterized nested workflows" item, since input defaults and override semantics designed here would directly apply to nested workflow parameterization. The `--input-file` complement handles the power-user case (CI pipelines, repeated runs with the same parameters) without complicating the interactive flow.

## 2. Execution Plan Preview (`orc start --dry-run`)

**Impact:** High
**Effort:** Medium
**Description:** Add a `--dry-run` flag to `orc start` that resolves the full execution plan without actually running any nodes. The output should show: (a) the resolved DAG with execution order and parallelization groups, (b) all template variables resolved to their concrete values (inputs, builtins like `{{workspace}}`, `{{repo_root}}`), (c) which nodes are conditional (`when:` guards) and whether they would execute given the current inputs, (d) which nodes have loops and their configuration, and (e) any validation warnings (unused inputs, unreachable nodes, cycles). For workflows with `depends_on` chains, display the critical path. Format as a structured text table or tree in the terminal. This serves as both a debugging tool for workflow authors and a confidence check before committing to a long-running execution.

**Rationale:** Workflow authoring today is a write-run-debug cycle: you write YAML, run it, wait for execution (potentially minutes), and discover issues only when a node fails. `orc validate` catches syntax errors but nothing about execution semantics — it cannot tell you that your `when:` guard references a variable that won't exist, or that your DAG has an unintended serial bottleneck. A dry-run preview collapses the feedback loop from minutes to milliseconds. This is especially valuable as workflows grow in complexity with nested workflows, conditional branches, and loops. Every serious workflow tool (Airflow, GitHub Actions, Terraform) provides plan/preview functionality because the cost of a failed multi-minute run compounds rapidly during authoring. The implementation can reuse the existing DAG resolver and template engine — it just stops before dispatching to providers.

## 3. Rich Terminal Status with DAG Progress Visualization

**Impact:** Medium
**Effort:** Medium
**Description:** Redesign `orc status <run-id>` to display a visual representation of the workflow DAG in the terminal, with each node showing its current state (pending, running, completed, failed, skipped), elapsed time, and dependency connections. Use box-drawing characters or a simple ASCII graph layout. For active runs, support a `--watch` / `--follow` flag that refreshes the display periodically (polling the SQLite database), showing nodes transition from pending to running to completed in real-time. Color-code node states (green=completed, yellow=running, red=failed, dim=pending/skipped). For completed runs, show the critical path highlighted and total wall-clock vs. sum-of-node-times to illustrate parallelization savings. This transforms `orc status` from a flat text dump into an at-a-glance operational view.

**Rationale:** Iteration 1 identified "CLI silence during execution" as the top UX gap, and proposed node lifecycle events in `orc start`. This idea takes a complementary approach: rather than changing `orc start`'s output model (which requires engine API changes as noted in the Iteration 1 task selection), it enriches the existing `orc status` polling path that already reads from the shared SQLite database. A `--watch` flag on `orc status` gives users real-time visibility without any engine changes — it simply re-queries the store on an interval. This is architecturally simpler and serves the same user need (knowing what's happening during a run) via a different interaction model (`orc start` in one terminal, `orc status --watch` in another). It also directly previews what the roadmap's "local web server" dashboard would show, letting the team iterate on the information design in the terminal before building the browser UI.
