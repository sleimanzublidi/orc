## Debate Summary

Verified all six ideas against the actual codebase. Key findings:

- **NodeDispatcher batch-sequential pattern**: Confirmed. TODO(M11) at line 76 acknowledges it. Real performance impact but architectural change is large.
- **Cross-process cancellation gap**: Confirmed but partially overstated. `CancellationHandler` does update DB and kill tmux sessions; the gap is specifically non-interactive process orphaning. Fix requires schema migration + signal handling — non-trivial.
- **LocalizedError absence**: Confirmed as a **live bug**. All 5 error types have rich `CustomStringConvertible` messages but no `LocalizedError`. `error.localizedDescription` is used for DB persistence in 5+ places in NodeDispatcher, producing opaque Foundation-default strings like `"The operation couldn't be completed. (ProviderError error 0.)"`. Found **17** `try?` calls silently discarding store write failures (not 8 as claimed). ConfigManager silent fallback confirmed.
- **Bundled templates**: Confirmed all three are single-node. Multi-node template is trivial to add but users are few during active development.
- **CLI silence during execution**: Confirmed. `orc start` prints one line before and one line after. No per-node events.
- **Actionable error messages**: Pattern proven by `OrcDirectory.require()` but "~10 error sites" is vague scope.

Scoring criteria applied strictly: product ideas that advance the product direction score highest on Value; engineering ideas score high only if they deliver measurable user-facing benefit. Feasibility = single focused commit. Safety = regression risk.

## Rejected Ideas

### Real-Time Execution Feedback (Product #1)
**Score:** Value 4 x Feasibility 2 x Safety 3 = **24**
**Reason:** Real problem, but requires changing the Engine API surface (currently fire-and-forget `start() -> Run`). Needs event streaming model (AsyncSequence/delegate), CLI consumer integration, and careful design to avoid coupling CLI concerns into the engine. Not a single-commit task. The design spec's "web server readiness" constraint means this event model needs to serve multiple consumers — getting it right matters more than getting it fast.

### Actionable Error Messages (Product #2)
**Score:** Value 4 x Feasibility 3 x Safety 4 = **48**
**Reason:** Genuine UX improvement but scope is underspecified. "~10 error sites" needs enumeration. Post-init messaging is trivial; early input validation requires adding a validation pass to `StartCommand` that checks required inputs against the parsed workflow before calling `engine.start()`. Recovery hints per error type means touching multiple commands. Good idea, but the vague scope makes it risky for a single focused commit. Better as a follow-up after error persistence is fixed — no point adding recovery hints if the errors themselves are opaque in the DB.

### Multi-Node Workflow Template (Product #3)
**Score:** Value 3 x Feasibility 5 x Safety 5 = **75**
**Reason:** Trivially implementable (one YAML file) and zero regression risk. However, Value is capped at 3: the product is in active development with few external users. The self-improve.yaml already demonstrates multi-node DAGs in practice. A bundled template matters when there's an onboarding funnel to optimize — that's premature right now. Additionally, demonstrating `loop:` and evaluators in a template requires the evaluator to actually work end-to-end, which adds hidden testing burden. Decent option but outscored.

### Streaming DAG Dispatch (Engineering #1)
**Score:** Value 3 x Feasibility 1 x Safety 2 = **6**
**Reason:** The batch-sequential limitation is real and acknowledged (TODO(M11)), but the fix is a significant architectural change. Restructuring TaskGroup dispatch with incremental result processing under Swift 6 strict concurrency requires careful state management (Mutex-protected struct or AsyncStream coordinator). The nested workflow parallelism concern (independent `maxParallelNodes` caps) adds further complexity. High risk of subtle concurrency bugs. This is a multi-commit, multi-test effort — not appropriate for a single iteration.

### Cross-process Cancellation + Signal Handling (Engineering #2)
**Score:** Value 3 x Feasibility 2 x Safety 2 = **12**
**Reason:** Correctness issue that users would hit, but the fix spans multiple subsystems: schema migration (new PID column), ProcessRunner changes (write PID to store), CancellationHandler changes (read PIDs and send signals), and SIGINT/SIGTERM handler installation in StartCommand. Signal handling in Swift is notoriously tricky (re-entrant signal handlers, async contexts). The PID-in-store approach is sound in principle but introduces a new cross-process coordination pattern that needs thorough testing. Not a single commit.

## Selected Task

### `LocalizedError` Conformance + Structured Error Persistence
**Source:** Engineering Ideas #3
**Priority:** P0
**Score:** Value 4 x Feasibility 5 x Safety 5 = **100**

**Description:** Fix a live bug where all persisted error messages are opaque Foundation-default strings. Add `LocalizedError` conformance to all 5 error types (ParserError, StoreError, ProviderError, EngineError, TemplateError) so `error.localizedDescription` returns the rich, structured messages already defined in `CustomStringConvertible`. Replace 17 `try?` store write calls in NodeDispatcher with `do/catch` + logging to prevent silent database state corruption. Add warning log in ConfigManager when config YAML fails to parse.

**Why this wins:** This is the rare engineering fix that is simultaneously high-value, trivially implementable, and zero-risk:
- **User-facing bug**: `orc logs <runID>` currently shows `"The operation couldn't be completed. (ProviderError error 0.)"` instead of `"Process failed: 'claude --prompt ...' exited with code 1 — stderr: API rate limit exceeded"`. Every user who hits an error gets useless diagnostic output.
- **Blocks other ideas**: Actionable Error Messages (Product #2) is pointless if the errors can't even be read in the DB. Real-Time Execution Feedback (Product #1) would inherit the same opaque error strings. This fix is a prerequisite.
- **Tiny change, huge surface area**: ~5 lines per error type (25 total for LocalizedError). The `try?` cleanup is mechanical. ConfigManager logging is a one-liner. All additive — no behavioral changes, no API surface changes, no schema migrations.

**Implementation notes:**
1. Add `LocalizedError` conformance to each error type via extension (e.g., `extension ParserError: LocalizedError { var errorDescription: String? { description } }`). Files: `ParserError.swift`, `StoreError.swift`, `ProviderError.swift`, `EngineError.swift`, `TemplateError.swift`.
2. In `NodeDispatcher.swift`, replace each `try?` store call with `do { try await store... } catch { logger.error("Failed to persist ...: \(error)") }`. Execution must not fail due to store issues — the catch logs but does not rethrow.
3. In `ConfigManager`, replace `try? Yams.load(yaml:)` with `do/catch` that logs `logger.warning("Failed to parse config at \(path), using defaults: \(error)")`.
4. Add tests: trigger a known error (e.g., invalid agent name), verify the persisted `NodeExecution.errorMessage` contains the structured detail rather than the Foundation default.
