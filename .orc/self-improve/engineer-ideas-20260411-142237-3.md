# Engineering Ideas — 2026-04-11  (Iteration 3)

## 1. Extract a Generic Retry Executor to Replace 4 Duplicated Retry Loops

**Impact:** High
**Risk:** Low

**Description:** The identical retry-with-delay loop is copy-pasted across four locations:

1. `NodeDispatcher.executeSingleNode` (lines ~377-419) — retries a single node execution
2. `NodeDispatcher.executeInteractiveNode` (lines ~469-506) — retries interactive node execution
3. `NodeDispatcher.executeLoopNode` (lines ~545-561) — retries loop node execution
4. `LoopHandler.executeIterationWithRetry` — retries individual loop iterations

All four follow the exact same structure:

```swift
for attempt in 1...maxAttempts {
    do {
        let result = try await executeWork()
        return result
    } catch {
        if attempt == maxAttempts { throw error }
        try await Task.sleep(for: .seconds(delaySeconds))
    }
}
```

This is ~20 lines per site × 4 sites = ~80 lines of duplicated control flow. More critically, each copy subtly diverges in how it logs attempts, records failures, and handles the final error — making it hard to reason about retry semantics globally.

**Proposed fix:** Extract a generic, `@Sendable`-safe retry executor as an internal utility in the Engine module:

```swift
func withRetry<T: Sendable>(
    maxAttempts: Int,
    delay: Duration,
    operation: @Sendable () async throws -> T
) async throws -> T
```

Each call site reduces to a single `try await withRetry(maxAttempts: node.retry.maxAttempts, delay: .seconds(node.retry.delaySeconds)) { ... }` invocation. The helper centralizes attempt counting, delay logic, and the final rethrow. This also creates a single point for future enhancements — exponential backoff, jitter, or per-error-type retry decisions — without touching 4 call sites.

**Rationale:** This is a textbook DRY violation with real maintenance cost. The subtle divergences between the four copies are not intentional — they're drift from independent edits. A single retry executor ensures consistent behavior and reduces the cognitive load when reasoning about retry semantics. The risk is minimal because the helper is a pure control-flow utility with no state or side effects.

**Baseline estimate:**
- *Current:* ~80 lines of retry boilerplate across 4 sites. Adding a new retry capability (e.g., exponential backoff) requires editing 4 methods.
- *Expected:* 4 call sites reduced to ~1 line each + ~15-line helper. New retry features require editing 1 method.
- *How to measure:* Line count before/after. Verify identical test coverage: existing retry tests in `NodeDispatcherTests` and `LoopHandlerTests` should pass without modification, confirming behavioral equivalence.

## 2. Introduce a Shared OrcTestSupport SPM Target to Deduplicate Test Infrastructure

**Impact:** Medium
**Risk:** Low

**Description:** Test fakes are duplicated across module boundaries:

- `FakeProcessRunner` exists in both `Core/Engine/Tests/Fakes/` and `Core/Providers/Tests/Fakes/` — two independent copies of the same type, each conforming to `ProcessRunning`.
- `FakeTmuxProvider` exists in both `Core/Engine/Tests/Fakes/` and `Core/Providers/Tests/Fakes/` — two independent copies conforming to `TmuxProviding`.

When a protocol method signature changes (e.g., adding a parameter to `ProcessRunning.run()`), both copies must be updated independently. If one is missed, only that module's tests fail — and the fix is a mechanical copy of what the other module already did. This is the kind of silent maintenance burden that compounds as the project grows.

Additionally, handler construction logic in `WorkflowEngine.start()` (lines 257-300) and `WorkflowEngine.resume()` (lines 360-402) builds identical `EvaluatorRunner → InteractiveHandler → LoopHandler → NodeDispatcher` chains. While not a test issue per se, the pattern suggests that test setup for integration-level Engine tests also duplicates this chain construction.

**Proposed fix:** Add an `OrcTestSupport` library target to `Package.swift` with `type: .dynamic` to avoid linker duplication:

```swift
.target(
    name: "OrcTestSupport",
    dependencies: ["OrcModels"],
    path: "Core/TestSupport/Source"
)
```

Move all shared fakes (`FakeProcessRunner`, `FakeTmuxProvider`, `FakeAgentProvider`, `FakeWorkflowStore`, `FakeWorkflowParser`) into this target. Each module's test target depends on `OrcTestSupport` instead of maintaining its own fakes. Mark fakes as `public` within this target so they're accessible from all test targets.

This also establishes a pattern: as new protocols are added (e.g., `ProviderRegistering` from Iteration 2's idea #3), their fakes go directly into `OrcTestSupport` — one copy, shared everywhere.

**Rationale:** The duplication is small today (2 fake types × 2 modules) but grows linearly with both protocol count and module count. A shared test support target is standard practice in multi-module SPM projects and has near-zero risk — it changes no production code, only test infrastructure. The cost of NOT doing this is that every protocol change requires N updates to N copies of each fake, with N being the number of modules that test against that protocol.

**Baseline estimate:**
- *Current:* 2 duplicated fakes, ~100 lines duplicated total. Protocol signature change requires updating 2 copies.
- *Expected:* 0 duplicated fakes. Protocol signature change requires updating 1 copy.
- *How to measure:* `grep -r "class FakeProcessRunner" Orc/` should return exactly 1 result after migration (vs. 2 today). All tests pass with no behavioral changes.

## 3. Replace Fetch-Modify-Write with Targeted Column Updates in WorkflowStore

**Impact:** Medium
**Risk:** Medium

**Description:** `WorkflowStore` implements state transitions by fetching the full record, modifying a single field in Swift, and writing the entire record back. This pattern appears in at least 5 methods:

- `updateRunStatus(runID:status:)` — fetches full `Run`, changes `.status`, writes all 9 columns back
- `updateRunOutput(runID:output:)` — fetches full `Run`, changes `.output`, writes all 9 columns back
- `updateRunWorkspacePath(runID:path:)` — fetches full `Run`, changes `.workspacePath`, writes all 9 columns back
- `updateNodeExecution(id:status:output:errorMessage:endedAt:)` — fetches full `NodeExecution`, modifies up to 4 fields, writes all 11 columns back
- `updateNodeExecutionIteration(id:iteration:)` — fetches full `NodeExecution`, changes `.iteration`, writes all 11 columns back

Each of these round-trips more data than necessary and requires the calling code to reconstruct the full record. More importantly, this pattern is fragile when columns are added: if a new column is added to the `runs` table (e.g., `pid` from Iteration 1's cancellation idea, or `progress_percent` for a future streaming feature), every update method must be verified to correctly preserve the new column — even methods that don't logically touch it.

**Proposed fix:** Replace the fetch-modify-write pattern with targeted SQL updates via GRDB's column-specific update API:

```swift
func updateRunStatus(runID: String, status: RunStatus) async throws {
    try await db.write { db in
        guard var run = try Run.fetchOne(db, key: runID) else {
            throw StoreError.recordNotFound(table: "runs", id: runID)
        }
        run.status = status
        try run.update(db, columns: [Run.Columns.status])
    }
}
```

GRDB's `update(db, columns:)` generates `UPDATE runs SET status = ? WHERE id = ?` — touching only the specified column. This is both more efficient (less data in the SQL statement) and self-documenting (the update method explicitly declares which columns it modifies). For methods that update multiple columns (like `updateNodeExecution`), list all modified columns: `try execution.update(db, columns: [.status, .output, .errorMessage, .endedAt])`.

**Rationale:** The current pattern works correctly today but is a maintenance trap. Every schema migration that adds a column to `runs` or `node_executions` creates a silent dependency: all fetch-modify-write methods must correctly handle the new column, even though they don't logically involve it. With targeted column updates, adding a new column requires zero changes to existing update methods — only methods that actually modify the new column need to reference it. This directly supports the roadmap items (parameterized nested workflows, output streaming) that will likely add columns to existing tables.

**Baseline estimate:**
- *Current:* 5 update methods each write all columns (9-11 per table). Adding a new column to `runs` requires auditing all 3 `updateRun*` methods.
- *Expected:* Each update method writes only 1-4 columns. Adding a new column to `runs` requires zero changes to existing update methods.
- *How to measure:* Enable GRDB query tracing (`db.trace { print($0) }`) and compare SQL output before/after. Before: `UPDATE runs SET id=?, workflow_file=?, status=?, ...all columns... WHERE id=?`. After: `UPDATE runs SET status=? WHERE id=?`. Verify with existing `WorkflowStoreTests` — all should pass unchanged.
