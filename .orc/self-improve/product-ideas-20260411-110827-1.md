# Product Ideas — 2026-04-11  (Iteration 1)

## 1. Real-Time Execution Feedback During `orc start`

**Impact:** High
**Effort:** Medium
**Description:** After printing "Running workflow X", `orc start` goes completely silent until the entire DAG completes. For multi-node workflows running for minutes, users have no idea whether the tool is working or hung. Add node lifecycle events to stdout: print a line when each node starts (`[running] plan...`), completes (`[completed] plan (12s)`), or fails (`[failed] deploy: exit code 1`). Show elapsed time on completion. For single long-running nodes, consider a minimal spinner or periodic heartbeat. This is distinct from the roadmap's "agent-level output streaming" (which is about piping agent stdout) — this is about DAG-level progress visibility, which is achievable without touching providers at all.

**Rationale:** This is the single biggest UX gap for any user running a non-trivial workflow. The information already exists internally (the engine dispatches and completes nodes) — it just needs to surface to the CLI layer. Node start/complete events with elapsed time would transform the experience from "is this broken?" to "I can see exactly what's happening." It also serves as a natural foundation for the roadmap's streaming and web-server features, since the same event model would feed all three consumers.

## 2. Actionable Error Messages and Recovery Guidance

**Impact:** High
**Effort:** Low
**Description:** Most error messages tell the user *what* went wrong but not *what to do next*. Three specific improvements would cover the highest-friction moments:

1. **Post-`orc init` next steps.** After successful init, print: "Try `orc start hello-shell` to verify your setup. Run `orc catalog` to see available workflows." Also list the 3 bundled workflow names that were installed.
2. **Recovery hints on failure.** When a run fails, append: "Use `orc logs <id> --node <node-id>` for details, or `orc resume <id>` to retry from the failure point." When `orc init` hits `projectAlreadyExists`, suggest `orc init --update`. When a run or provider is not found, suggest `orc list` or list available providers respectively.
3. **Early validation of required inputs.** Before starting execution, check that all `required: true` workflow inputs have been provided. Currently, missing inputs surface deep in the engine as cryptic `TemplateError.unresolvedVariable` messages. A clear "Missing required input 'repo_path'. Provide it with --input repo_path=<value>" at the CLI layer would prevent confusion.

**Rationale:** The `OrcDirectory.require()` message ("No .orc/ directory found. Run 'orc init' to create one.") proves the pattern works — it's the single most helpful error in the codebase. Extending this pattern to the other ~10 error sites is low-effort, high-return work. New users hit these errors in their first 5 minutes; recovery guidance is the difference between "I figured it out" and "I gave up."

## 3. Multi-Node Workflow Template Demonstrating Core Features

**Impact:** Medium
**Effort:** Low
**Description:** The three bundled workflow templates (`hello-shell`, `ask-claude`, `review-file`) are all single-node workflows. None demonstrate `depends_on`, automatic parallelization, `when:` conditionals, loops, or interactive nodes — the features that differentiate Orc from a simple script runner. Add a `plan-and-implement.yaml` template (2-3 nodes with dependencies) that shows: (a) a planning node whose output feeds an implementation node via `depends_on`, (b) a `when:` guard on an optional review node, and (c) commented-out examples of `loop:` and `interactive:` configuration. This template would be the primary learning vehicle for Orc's DAG model.

**Rationale:** Bundled templates are the de facto tutorial — most users will look at existing workflows before writing their own. Right now, those templates teach "Orc runs a single agent with inputs," which undersells the product dramatically. A multi-node template would answer the question every new user has: "Why should I use this instead of a shell script?" The effort is minimal (one YAML file with good comments), but the impact on first-time comprehension of the product's value proposition is significant. This also fills the empty evaluators directory gap by including a simple evaluator example alongside the workflow.
