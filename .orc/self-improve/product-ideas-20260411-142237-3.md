# Product Ideas — 2026-04-11  (Iteration 3)

## 1. Guided Workflow Scaffolding (`orc new`)

**Impact:** High
**Effort:** Medium

**Description:** Add an interactive `orc new` command that walks users through creating a workflow YAML file step by step. The flow: (1) ask for workflow name and description, (2) iteratively add nodes — for each node, prompt for name, agent type (offering the list from `orc config` or defaults: `claude-code`, `shell`), prompt/command text, and dependencies (showing previously added nodes as options), (3) ask about inputs the workflow should accept, (4) generate a well-commented YAML file with the correct schema, including placeholder `when:` guards and `output:` mappings as comments showing available syntax. For non-interactive contexts, support `orc new --from-template <name>` that copies a catalog template and opens it for editing. The generated file should include inline comments explaining each field (e.g., `# depends_on: list node names that must complete before this node runs`).

**Rationale:** The current onboarding path is: read README, look at bundled templates, copy-paste-modify YAML. This works for experienced CLI users but creates a steep learning curve for the primary use case — someone who wants to chain AI agents together but doesn't know Orc's YAML schema. Every successful CLI tool with a non-trivial config format provides scaffolding (`npm init`, `cargo init`, `gh repo create`). The information needed to generate valid YAML already exists in the parser's validation logic — this command just inverts it from "validate after the fact" to "guide upfront." This also naturally complements the Iteration 2 dry-run idea: `orc new` creates the workflow, `orc start --dry-run` validates it, `orc start` runs it — a complete authoring loop.

## 2. Run Completion Hooks (`--on-complete` and `--notify`)

**Impact:** High
**Effort:** Low

**Description:** Add two flags to `orc start`: (1) `--notify` sends a macOS notification (via `NSUserNotification` or `osascript -e 'display notification ...'`) when the run completes or fails, showing the run ID, final status, and elapsed time; (2) `--on-complete <command>` executes an arbitrary shell command after the run finishes, with environment variables `ORC_RUN_ID`, `ORC_STATUS` (completed/failed/cancelled), `ORC_ELAPSED_SECONDS`, and `ORC_WORKFLOW_NAME` injected. Examples: `orc start deploy.yml --notify`, `orc start ci.yml --on-complete "slack-notify '#deploys' 'CI run $ORC_STATUS'"`. The notification/hook fires regardless of success or failure. For `--notify`, use different notification sounds or urgency levels for success vs. failure.

**Rationale:** Multi-node workflows with AI agents routinely take 2-10+ minutes. During that time, users either stare at the terminal or switch tasks and forget to check back. This is the exact problem that CI systems solved with Slack/email notifications decades ago. The implementation is minimal — both hooks fire after `engine.start()` returns in `StartCommand`, requiring ~30 lines of code in the CLI layer with zero engine changes. `--on-complete` also enables composing Orc into larger automation pipelines (trigger downstream jobs, update dashboards, send alerts) without needing the planned web server. This is the cheapest feature that transforms Orc from a synchronous tool into an async-friendly one.

## 3. Structured Output Access (`orc output <run-id>`)

**Impact:** Medium
**Effort:** Medium

**Description:** Add an `orc output` command that displays the captured outputs of a completed run in a structured, machine-readable format. `orc output <run-id>` shows each node's output as a labeled section. `orc output <run-id> --node <node-id>` shows a single node's output. `orc output <run-id> --json` emits all outputs as a JSON dictionary keyed by node ID, suitable for piping into `jq` or consuming from scripts. `orc output <run-id> --node <node-id> --raw` emits just the raw output text with no formatting, for direct piping. Currently, to see what a workflow produced, users must run `orc logs <run-id>`, scan through interleaved log entries from all nodes, and mentally separate "what the agent did" from "what it produced." The output command provides a clean read path focused on results rather than process.

**Rationale:** Workflows exist to produce results — plans, code, reviews, reports. But Orc's current interface is optimized for observing execution (status, logs) rather than consuming results. This matters most for automation chains: if a planning workflow produces a task list that feeds into an implementation workflow, the user currently has to manually extract the output from logs. `orc output <run-id> --node plan --raw` would make this a one-liner. The `--json` format also enables scripting patterns like `orc start plan.yml && TASKS=$(orc output $LAST_RUN --node plan --raw) && orc start implement.yml --input tasks="$TASKS"`. The data already exists in the store's `NodeExecution.output` field — this command is primarily a formatting and access layer over existing persistence. This also serves as a stepping stone toward the roadmap's parameterized nested workflows, since structured output access is the prerequisite for programmatic workflow composition.
