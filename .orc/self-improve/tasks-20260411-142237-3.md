# Task Selection — 2026-04-11  (Iteration 3)

## Debate Summary

Evaluated 3 product ideas and 3 engineering ideas against codebase evidence. Key findings:

- **Engineering claims partially fabricated:** Eng #3 cites 5 store update methods, but `updateNodeExecutionIteration` does not exist — only 4 methods are real. This weakens trust in the analysis.
- **Engineering ideas are internal refactors with no user-facing value.** The retry dedup (Eng #1) is real but affects ~80 lines across 4 sites — not a maintenance emergency. The test support target (Eng #2) involves only 2 fakes that have *diverged* (different handler signatures, different recording strategies), making unification harder than the idea suggests. The column update fix (Eng #3) is correct but theoretical — the 4 existing update methods work fine and schema changes are infrequent.
- **Product #2 (Completion Hooks) dominates** on all three axes: high user value (transforms Orc from sync-only to async-friendly), high feasibility (~50 lines in CLI layer, zero engine changes), and near-zero risk (fires after `engine.start()` returns, no mutation of existing behavior).

## Rejected Ideas

### Guided Workflow Scaffolding — `orc new` (Product #1)
**Reason:** Real gap, but Medium effort — interactive prompting, template catalog, inline comment generation. Not achievable in a single focused commit. The project already has `orc init` for project setup and bundled templates in the catalog. Scaffolding is UX polish, not a capability gap. Defer until core workflow execution features are complete.

### Structured Output Access — `orc output` (Product #3)
**Reason:** Genuine need for automation chains, and the data already exists in `NodeExecution.output`. However, this requires a new command file, argument parsing for `--node`, `--json`, and `--raw` flags, plus multiple output formatters. That's Medium effort — likely 150-200 lines across command + formatting logic. Solid P1 candidate for a future iteration but not the highest-impact single commit available now.

### Extract Generic Retry Executor (Engineering #1)
**Reason:** The 4 duplicated retry loops are confirmed and real. However, extracting a `withRetry` helper delivers zero user-facing value. The subtle differences across sites (logging in #1 only, store updates in #1/#2 only, throw-vs-tuple in #4) mean the extraction isn't as clean as proposed — you'd need the caller to handle pre/post-loop concerns, reducing the actual dedup benefit. The 80 lines of boilerplate are annoying but not causing bugs. Value: 2, Feasibility: 4, Safety: 4 = 32.

### Shared OrcTestSupport SPM Target (Engineering #2)
**Reason:** Only 2 fakes are duplicated, and they've *diverged meaningfully* — `FakeProcessRunner` in Engine tracks full call histories via arrays while the Providers version uses scalars; `FakeTmuxProvider` uses countdown counters in Engine vs. explicit session sets in Providers. Unifying them requires designing a superset API that satisfies both test suites, modifying `Package.swift` structure, and updating all consuming tests. This is more work and risk than the idea acknowledges, for a problem that currently affects exactly 2 types. Value: 1, Feasibility: 2, Safety: 3 = 6.

### Targeted Column Updates in WorkflowStore (Engineering #3)
**Reason:** The fetch-modify-write pattern is confirmed in 4 methods (not 5 — `updateNodeExecutionIteration` was fabricated). The fix is mechanically correct (GRDB's `update(db, columns:)` exists and is unused). But the "maintenance trap" argument is theoretical — schema migrations that add columns are infrequent, and auditing 3-4 update methods during a migration is trivial. No user-facing benefit whatsoever. Value: 1, Feasibility: 4, Safety: 4 = 16.

## Selected Task

### Run Completion Hooks (`--notify` and `--on-complete`)
**Source:** Product Ideas #2
**Priority:** P1
**Description:** Add two flags to `orc start`: `--notify` sends a macOS notification (via `osascript`) when the run completes or fails, showing run ID, status, and elapsed time. `--on-complete <command>` executes an arbitrary shell command after the run finishes with environment variables `ORC_RUN_ID`, `ORC_STATUS`, `ORC_ELAPSED_SECONDS`, and `ORC_WORKFLOW_NAME` injected. Both fire regardless of success or failure.
**Implementation notes:**
- All changes in `StartCommand.swift` — zero engine modifications required.
- Add `@Flag var notify: Bool` and `@Option var onComplete: String?` to `StartCommand`.
- After `engine.start()` returns, compute elapsed time and determine final status from the run result.
- For `--notify`: shell out to `osascript -e 'display notification "..." with title "Orc"'` with different titles for success vs failure. Use `ProcessRunner` or a simple `Process` call since this is CLI-layer code.
- For `--on-complete`: execute the command string via `/bin/sh -c` with the four env vars set in the process environment.
- Estimated ~50-60 lines of new code, single commit.
- Test: verify flag parsing with unit tests; notification and shell execution can be tested by checking the constructed command/env rather than actually executing (mock the shell call).
