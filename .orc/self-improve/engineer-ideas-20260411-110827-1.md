# Engineering Ideas — 2026-04-11  (Iteration 1)

## 1. Streaming DAG Dispatch (replace batch-sequential pattern)

**Impact:** High
**Risk:** Medium

**Description:** The current `NodeDispatcher` (at `Core/Engine/Source/NodeDispatcher.swift:56-188`) uses a batch-sequential pattern: it dispatches all ready nodes as a batch via `withThrowingTaskGroup`, waits for the **entire batch** to complete, then checks for newly unblocked nodes. This means if node A finishes in 1 second but co-batched node B takes 60 seconds, any node depending only on A waits the full 60 seconds. The TODO at line 76 (tagged `TODO(M11)`) acknowledges this.

The fix: process `TaskGroup` results incrementally inside the `for try await result in group` loop. As each node completes, immediately update `nodeStatuses`/`nodeOutputs` and check dependents for readiness. Newly ready nodes are added to the group via `group.addTask` within the same iteration. This requires the mutable dispatch state (`nodeStatuses`, `nodeOutputs`, `pendingNodes`) to be accessible inside the `TaskGroup` closure -- likely via an `AsyncStream`-fed coordinator or by restructuring the state into a `Mutex`-protected struct (available in Swift 6.1's `Synchronization` module), since the current snapshot-based approach (`snapshotOutputs`/`snapshotStatuses` at lines 92-93) was specifically designed to satisfy `Sendable` requirements.

A secondary issue amplified by this change: nested workflows each create their own `NodeDispatcher` with an independent `maxParallelNodes` cap (line 550-774), so deeply nested workflow trees can exceed the intended global parallelism. A streaming dispatcher should incorporate a shared semaphore or global concurrency limiter.

**Rationale:** This is the single highest-leverage performance optimization for the engine. Workflows with heterogeneous node durations (common in real use -- e.g., a quick shell validation followed by a long Claude Code generation) pay a significant latency penalty from the batch barrier. The fix also eliminates the O(N*D) full-pending-set rescan per iteration (`isNodeReady` called for every pending node at line 60-65), replacing it with targeted checks of only the completed node's dependents.

**Baseline estimate:**
- *Current:* A 3-node DAG (A: 2s, B: 60s, C depends on A) takes ~62s. Measured by: `orc start` with `--input` timing, or by adding `os_signpost` / structured logging around batch boundaries in `NodeDispatcher`.
- *Expected:* Same DAG takes ~4s (A runs, C starts immediately after A, both finish before B). Improvement scales with DAG width and duration variance.
- *How to measure:* Add timestamps to the `NodeExecution` records (start/end per node are already stored). Compare wall-clock run duration vs. critical-path duration (sum of the longest dependency chain). The ratio should approach 1.0 with streaming dispatch.

## 2. Cross-process cancellation and signal handling

**Impact:** High
**Risk:** Medium

**Description:** Two related reliability gaps make cancellation unreliable:

**(a) `orc cancel` from another terminal does not kill subprocesses.** `WorkflowEngine.cancel()` (at `Core/Engine/Source/WorkflowEngine.swift:446-448`) looks up the in-flight `Task` from the `runningTasks` dictionary and calls `.cancel()` on it. But `runningTasks` is an in-memory dictionary on the actor instance. When `orc cancel <runID>` runs from a different terminal, it creates a **new** `WorkflowEngine` instance with an empty `runningTasks` -- so cooperative cancellation never fires. Only the `CancellationHandler` runs, which updates database records and kills tmux sessions, but standard (non-interactive) processes spawned by `ProcessRunner` continue running as orphans.

**(b) No signal handler on the engine process.** If the user presses Ctrl+C during `orc start`, the process is killed without cleanup. The run remains stuck as `.running` in the database forever. `orc resume` will refuse it (wrong status). The user must manually fix the database or run `orc purge`.

**Proposed fix:**

For (a): When `ProcessRunner` starts a subprocess, write the child PID to the `NodeExecution` record in the store (add a `pid` column). `CancellationHandler` then reads PIDs from the store and sends `SIGTERM`/`SIGKILL` directly, rather than relying on cooperative task cancellation. This works cross-process because the store is shared (SQLite WAL mode).

For (b): Install a `SIGINT`/`SIGTERM` handler (via `signal()` or `DispatchSource.makeSignalSource`) in the `orc start` command that calls `engine.cancel(runID:)` before exiting. This ensures database records are updated and subprocesses are terminated gracefully.

**Rationale:** These are correctness bugs, not optimizations. A user who runs `orc cancel` reasonably expects the workflow to actually stop. A user who presses Ctrl+C expects the system to clean up. Both expectations are violated today. The PID-in-store approach is robust because it uses the already-shared SQLite database as the coordination mechanism, avoiding the need for IPC or lock files.

**Baseline estimate:**
- *Current:* `orc cancel` from another terminal: subprocesses continue running (100% orphan rate for non-interactive nodes). Ctrl+C: run stuck as `.running` indefinitely.
- *Expected:* `orc cancel`: all subprocesses terminated within 5s (SIGTERM + SIGKILL grace period). Ctrl+C: run marked `.cancelled`, subprocesses terminated.
- *How to measure:* Start a workflow with a long-running shell node (`sleep 300`). From another terminal, run `orc cancel`. Verify with `ps aux | grep sleep` that the subprocess is gone. Check `orc status` shows `.cancelled`. Repeat with Ctrl+C.
